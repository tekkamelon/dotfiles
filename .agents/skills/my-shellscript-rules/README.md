# my-shellscript-rules README

このREADMEは、`SKILL.md`に記載した`my-shellscript-rules`の内容を日本語で解説する文書です。

## 概要

`my-shellscript-rules`は、シェルスクリプト作成時の実践的な規約をまとめたskillです。
主な目的は次の3点です。

- POSIX互換性の維持
- 可読性の確保
- 記述スタイルの一貫性

## 各セクションの解説

### 1. Purpose

シェルスクリプトを「移植しやすく、読みやすく、保守しやすい」状態で書くためのルール群であることを示しています。

### 2. Scope

- 明示的な指示がない限り、POSIX互換のシェルスクリプトを対象とします。
- 特定環境依存の機能ではなく、広く利用可能なコマンドやオプションを優先します。

### 3. Command Usage Rules

#### grep

- 正規表現が不要な場面では、固定文字列検索の`grep -F`を使用します。
- 意図が明確になり、エスケープの負担を減らせます。

#### awk

- `mawk`で動作する書き方を前提とします。
- GNU awk(gawk)固有拡張は使わず、移植性を優先します。

### 4. Argument Parsing Rules

- `getopts`は使用しません。
- `while [ "${#}" -gt 0 ]; do`で引数を逐次処理します。
- `case`で分岐し、`shift`で読み進めます。
- `-i value`のように値を取るオプションは、値不足を明示的に検証します。

この方式により、引数処理の流れを上から下へ追いやすくなります。

### 5. Variable Rules

- 変数展開は`"${name}"`形式を基本とします。
- 環境変数は`UPPER_SNAKE_CASE`、通常変数は`lower_snake_case`を使用します。

これにより、単語分割やパス展開の事故を減らし、変数の役割も判別しやすくなります。

### 6. Readability Rules

- 左から右、上から下へ読める構成を重視します。
- 不要な関数分割を避け、処理の流れを追いやすくします。
- `if`や`case`などの制御構文には適切な改行を入れます。
- 論理的なまとまりごとに空行を入れます。

### 7. Section Markers

関数定義や変数定義の範囲を明示するため、次のような区切りコメントを推奨します。

```sh
# ====== 関数の宣言 ======
# ...
# ====== 関数の宣言ここまで ======

# ====== 変数の宣言 ======
# ...
# ====== 変数の宣言ここまで ======
```

### 8. Comment Rules

- コメントは日本語で記述します。
- 文末を動詞で終えるのではなく、体言止めで簡潔に書きます。

例:

- 良い例: `# 文字列を出力`
- 避ける例: `# 文字列を出力する`

## Quick Checklistの使い方

`SKILL.md`末尾のチェックリストは、実装前後のセルフレビューに使います。
次を短時間で確認できます。

- コマンド選択(`grep -F`, mawk互換awk)
- 引数解析方式(`while + case + shift`, `getopts`不使用)
- 変数規約(命名, 展開)
- 可読性(改行, 空行, セクション区切り)
- コメント規約(日本語, 体言止め)

## 想定される利用方法

- 新規シェルスクリプト作成時のテンプレート規約として利用
- レビュー時の観点統一に利用
- 既存スクリプトのリファクタリング指針として利用
